# Phase 03.1: Lock Polymarket ratchet benchmark - Context

**Gathered:** 2026-04-08
**Status:** Ready for planning

<domain>
## Phase Boundary

Turn the newly improved Polymarket BTC strategy behavior into a ratcheted benchmark artifact and regression guard. This phase should preserve the current improvement, expose the intended signal fields consistently, and prevent future strategy tweaks from silently falling back below the new floor. It should not redesign the broader strategy stack or replace Phase 4 layered-position work.

</domain>

<decisions>
## Implementation Decisions

### Benchmark scope
- **D-01:** This urgent phase is BTC-only. It protects the newly improved Polymarket strategy behavior before broader multi-ticker strategy work continues.
- **D-02:** The ratchet benchmark window should match the actual historical Polymarket coverage window now driving the strategy, starting at `2025-08-08`, rather than a longer chart span where the strategy is flat because no Polymarket history exists.
- **D-03:** The benchmark should lock the improved history-aware strategy path, not the old raw aggregate skew path.

### Promotion floor
- **D-04:** The current promoted Polymarket floor is the post-improvement result already observed through `/api/chart` for `BTC-USD` with start `2025-08-08`: `ending_equity 9494.0`, `total_pnl -505.99`, `net_profit_pct -5.06`, `max_drawdown_pct 8.89`, `total_trades 3`.
- **D-05:** Future Polymarket changes should not be allowed to degrade below this promoted floor without an explicit baseline update. This is the ratchet to preserve.
- **D-06:** The guard should focus on the user-visible route contract and strategy summary, not just isolated helper math, so regressions are caught where the app actually consumes the strategy.

### Signal contract
- **D-07:** The strategy should continue using the relevance-weighted Polymarket signal built from near-spot strike weighting when snapshot context is available.
- **D-08:** The live Polymarket API should expose both raw and strategy-ready metrics: `skew_ratio` for transparency, plus `signal_skew_ratio` and `signal_source` for the actual decision path.
- **D-09:** If live spot lookup is unavailable, the strategy should fall back to the latest saved snapshot spot price rather than degrading unnecessarily to raw-skew behavior.

### Validation style
- **D-10:** Preserve the repo's existing benchmark pattern: fixture-backed route-level regression checks rather than ad hoc manual comparisons.
- **D-11:** This phase should produce an inspectable artifact in-repo, alongside a focused regression test that makes "got worse again" immediately visible.
- **D-12:** Planning should assume the implementation already exists locally; the phase work is to formalize, fixture, and document it rather than invent a second Polymarket redesign.

### the agent's Discretion
- Exact fixture file naming and placement, as long as it follows repo benchmark conventions.
- Whether the ratchet assertion is a strict exact-match floor or a bounded-threshold floor, as long as it prevents silent backsliding on the key summary metrics.
- Whether the live signal contract gets an additional focused route test or is covered through the broader benchmark harness.

</decisions>

<specifics>
## Specific Ideas

- The key improvement came from weighting Polymarket strikes by relevance to current BTC spot instead of letting far-out, high-volume strikes dominate the strategy.
- The current live-history result improved from roughly `-25.14% / 32.15% max drawdown / 1 trade` to roughly `-5.06% / 8.89% max drawdown / 3 trades`.
- The benchmark should make that improvement durable before any later strategy work starts layering entries/exits or expanding the ratchet logic further.

</specifics>

## Existing Code Insights

### Reusable Assets
- `routes/chart.py` already exposes a Polymarket summary through the same route shape the benchmark harness can pin.
- `tests/test_btc_benchmark_backtests.py` already demonstrates the route-fixture ratchet pattern this phase should likely copy.
- `tests/test_polymarket.py` already covers the new relevance-weighted signal internals, so this phase does not need to re-prove the low-level math from scratch.

### Established Patterns
- The repo prefers persisted benchmark fixtures checked through pytest rather than hidden runtime baselines.
- User-visible strategy behavior is validated through `/api/chart`, not only through internal helpers.
- Strategy contracts can expose richer fields while still keeping prior summary fields available for transparency.

### Integration Points
- The likely integration points are `tests/fixtures/`, a new focused benchmark test under `tests/`, and possibly a small helper script if fixture regeneration is needed.
- If the benchmark is route-driven, it should stay aligned with the current Polymarket-specific behavior in `routes/chart.py`.
- If the live signal contract also gets pinned, `routes/polymarket.py` should remain synchronized with the saved-history strategy semantics.

</code_context>
