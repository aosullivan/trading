# Phase 03.2: Lock Current Strategy Configuration Ratchet - Context

**Gathered:** 2026-04-08
**Status:** Ready for planning

<domain>
## Phase Boundary

Strengthen the current ratchet so the app cannot silently regress its present backtest configuration surface while later strategy work continues. This phase should freeze the current strategy selector, money-management defaults, and route-level strategy contract in deterministic artifacts and tests. It should not redesign the strategy logic itself or replace Phase 4 layered-position work.

</domain>

<decisions>
## Implementation Decisions

### Ratchet scope
- **D-01:** This urgent phase protects the current configuration contract, not just performance floors. It should freeze what the user can select today and how those selections map to `/api/chart`.
- **D-02:** The ratchet should cover three surfaces together: the backtest UI option ordering, the JavaScript default strategy selection, and the backend `strategies` payload contract.
- **D-03:** The strengthened ratchet should reuse existing frozen BTC and Polymarket fixtures so tests remain deterministic and do not depend on live Yahoo or live Polymarket responses.

### Config contract
- **D-04:** The strategy selector order matters and should be pinned as a product contract, especially the default `ribbon` entry and the presence of `corpus_trend` and `polymarket`.
- **D-05:** The money-management controls should be ratcheted as well: current sizing, stop, risk-cap, and compounding defaults/options are part of the user-facing behavior and should not drift silently.
- **D-06:** The JS default strategy should remain `ribbon` unless there is an intentional baseline promotion, so the ratchet should explicitly assert that constant.

### Route contract
- **D-07:** The route-level guard should use `/api/chart`, not internal helpers, because that is the surface the UI consumes.
- **D-08:** The backend ratchet should assert the exact expected strategy-key inventory and required payload fields for each strategy, while allowing individual strategies to keep their current buy-hold behavior.
- **D-09:** The route contract should be exercised on `BTC-USD` with frozen OHLCV plus frozen Polymarket history so both `corpus_trend` and `polymarket` are present in the same deterministic request.

### Promotion semantics
- **D-10:** This phase is a no-drift contract ratchet, not a new performance promotion. It should preserve the current configuration until a later phase intentionally changes the baseline.
- **D-11:** If a later change intentionally alters defaults, labels, or option ordering, it must update the machine-readable spec, docs, and the ratchet tests together in the same change.
- **D-12:** The inserted phase should hand control back to Phase 4 once the config contract is frozen; it should not expand into broader redesign or UI restyling work.

### the agent's Discretion
- Exact artifact naming, as long as it follows the repo's existing ratchet conventions.
- Whether the contract lives in one JSON spec or multiple smaller files, as long as the tests read one clear source of truth.
- Whether the route-level test also asserts order of backend strategy keys, as long as it clearly protects the expected inventory and user-visible compatibility.

</decisions>

<specifics>
## Specific Ideas

- Freeze the current strategy selector values and labels into a machine-readable spec so future additions or reorderings are always explicit.
- Freeze the current money-management defaults (`All-In`, `None`, `None`, `Per Trade`) and option lists because those settings define the present backtest experience.
- Use a deterministic `/api/chart` request on `BTC-USD` with frozen Polymarket history so the route contract includes both the main strategy stack and the currently improved Polymarket strategy.

</specifics>
