# Phase 1 Research: Build Trend-Following Knowledge Base

## Planning Goal
This phase should produce a source-cited, human-readable knowledge base distilled from every transcript in `audio/`, plus enough structure to let Phase 2 turn those principles into concrete strategy rules without rereading the corpus.

Core source context: `.planning/phases/01-build-trend-following-knowledge-base/01-CONTEXT.md`, `.planning/REQUIREMENTS.md`, `.planning/ROADMAP.md`, `.planning/STATE.md`, `.planning/codebase/STACK.md`, `.planning/codebase/STRUCTURE.md`, `.planning/codebase/CONVENTIONS.md`, `.planning/codebase/CONCERNS.md`, `scripts/transcribe_audiobook.py`.

## Corpus Shape and Extraction Implications
- `audio/` contains 75 transcript `.txt` files spanning opening credits, Chapter 1 through Chapter 73, and end credits. Filenames encode chapter order well enough to sort deterministically by path/name, satisfying `KB-01`.
- Transcript files are plain text generated by `scripts/transcribe_audiobook.py` via segment-level `faster_whisper` output joined with newlines, with no embedded timestamps or explicit chapter metadata beyond the filename.
- Sample files show two distinct content modes that the extractor must handle differently:
  - Narrative chapters with direct trend-following principles, e.g. `audio/Trend Following, 5th Edition - 02 - Chapter 1.txt`.
  - Margin-quote-heavy transcripts whose text starts with phrasing like “side margin quotations” and then lists short quotes and attributions, e.g. `audio/Trend Following, 5th Edition - 11 - Chapter 10.txt` and `audio/Trend Following, 5th Edition - 31 - Chapter 30.txt`.
- Planning implication: the phase needs a lightweight chapter classifier or extraction convention so margin-quote-heavy files are retained for citations and color, but do not dominate the synthesized strategy principles.

## Recommended Knowledge-Base Artifact
Use one canonical Markdown artifact plus one optional machine-readable index:
- Primary artifact: `.planning/phases/01-build-trend-following-knowledge-base/trend-following-knowledge-base.md`
- Optional sidecar: `.planning/phases/01-build-trend-following-knowledge-base/trend-following-knowledge-base.json`

Recommended Markdown schema:
- `# Overview`
- `## Corpus Coverage`
  - total files processed, skipped files with reasons, and chapter-order rule
- `## Source Index`
  - one row per transcript with filename, inferred chapter label, content mode (`narrative`, `margin-quotes`, `credits`, `mixed`), and short extraction notes
- `## Principle Catalog`
  - sections aligned to `KB-02`: entries, exits, position sizing, risk control, drawdown discipline, trend persistence, whipsaw handling, portfolio/market selection, regime assumptions
  - each principle should include:
    - `claim`: concise paraphrase
    - `evidence`: source refs by filename/chapter
    - `strategy implication`: how this narrows Phase 2 design
    - `confidence`: `high|medium|low`
    - `alternatives`: competing interpretations where transcript evidence conflicts
- `## Strategy Design Implications`
  - candidate rule families, non-negotiable constraints, and open questions Phase 2 must resolve

Recommended JSON sidecar shape:
```json
{
  "corpus": [{"file": "...", "chapter": 1, "mode": "narrative", "processed": true, "skip_reason": null}],
  "principles": [{
    "id": "risk-001",
    "category": "risk_control",
    "claim": "...",
    "sources": [{"file": "...", "chapter": 1, "evidence_type": "paraphrase"}],
    "strategy_implication": "...",
    "confidence": "high",
    "alternatives": []
  }]
}
```

Why this shape fits the codebase:
- Markdown aligns with the repo’s GSD planning docs and makes review easy in `.planning/`.
- A small JSON sidecar, if implemented, gives Phase 2 deterministic parsing without changing runtime app structure in `lib/`, `routes/`, `templates/`, or `static/js`.

## Deterministic Ingestion and Cleaning Concerns
- Sort transcript paths deterministically using filename order from `audio/`, not filesystem traversal order. Persist the exact ordered source list in the artifact’s `Source Index`.
- Derive chapter metadata from filenames, not transcript body text, because body text may contain OCR/ASR drift and repeated stock phrases.
- Preserve all transcript files in coverage accounting, but allow content-aware extraction modes:
  - `credits`: likely metadata-only, low strategy signal
  - `margin-quotes`: quote-heavy chapter supplements, useful as supporting context but lower weight for rule derivation
  - `narrative` or `mixed`: primary signal for principle extraction
- Text cleanup should be conservative and documented:
  - trim whitespace and normalize blank lines
  - fix obvious line-join artifacts only when deterministic
  - avoid rewriting punctuation or terminology aggressively because ASR quirks may matter for search traceability
- Do not depend on timestamps, speaker labels, or paragraph boundaries; `scripts/transcribe_audiobook.py` does not emit them.

## Citation and Source-Mapping Strategy
- Every principle in the KB should cite one or more transcript files using filename + inferred chapter number, e.g. `audio/Trend Following, 5th Edition - 02 - Chapter 1.txt`.
- Prefer paraphrased evidence summaries over long copied text to stay reviewable and avoid overfitting on ASR wording. If short direct quotes are included, keep them minimal and attach the source file immediately.
- Assign stable principle IDs (`entry-001`, `risk-002`, `portfolio-001`) so Phase 2 can reference KB conclusions unambiguously in implementation notes and tests.
- Keep a many-to-many relation:
  - one transcript file can support multiple principles
  - one principle can cite multiple transcript files
- Record source mode in the `Source Index` so Phase 2 can distinguish narrative evidence from margin-quote support.

## Implementation Pitfalls to Plan Around
- Margin-quote-heavy chapters can produce lots of inspirational or definitional snippets with weak direct strategy signal. Plan an extraction rule that separates “supporting quote” from “actionable trading principle.”
- Some transcript text likely contains ASR transcription errors and odd chapter labeling. The chapter filename should be treated as canonical ordering metadata, while body text should be interpreted cautiously.
- A manual-only synthesis pass risks missing corpus coverage. The phase plan should include an explicit coverage ledger and a final completeness check against all 75 transcript files.
- Over-structuring too early can force false precision. Keep principle fields small and allow `alternatives` plus `open_questions` so ambiguity survives into Phase 2 instead of being silently collapsed.
- Do not write implementation assumptions directly into production modules in `lib/` during Phase 1. Per `.planning/ROADMAP.md`, Phase 1 is an artifact-building phase; strategy code belongs in Phase 2.

## Phase 2 Handoff Contract
The Phase 1 artifact should hand Phase 2 a compact strategy brief that can be translated into code and tests:
- Candidate entry models implied by corpus evidence (breakout, trend confirmation, moving-average/channel families, or “price-only” constraints if strongly supported)
- Candidate exit models and stop logic
- Position sizing/risk budget constraints and drawdown behavior
- Market universe guidance and regime assumptions
- Explicit anti-requirements, especially “do not optimize/tweak during drawdowns” or “avoid prediction/fundamental filters” if those principles are repeatedly supported in `audio/`
- Open design choices requiring Phase 2 tradeoff decisions, each linked to principle IDs and transcript evidence

This handoff should be written so `02-01` can map KB principles into a strategy spec before touching `lib/technical_indicators.py`, `lib/backtesting.py`, or strategy-related UI code documented in `.planning/codebase/STRUCTURE.md`.

## Validation Architecture
There is a concrete way to verify artifact completeness and basic quality before Phase 2:
- Coverage validation:
  - enumerate all `audio/*.txt` files, compare against the artifact `Source Index`, and fail if any transcript is missing or duplicated
  - verify chapter ordering in the index matches deterministic filename sort
- Citation validation:
  - every principle has at least one source ref
  - every source ref points to an existing file under `audio/`
  - every `strategy implication` references at least one principle ID or lives under a principle entry
- Taxonomy validation:
  - each `KB-02` category has at least one synthesized principle or an explicit “no strong evidence found” note with citations
- Extraction-quality validation:
  - margin-quote-heavy files are marked in the source index and do not appear as the only evidence for all major principles
  - skipped files, if any, are listed with reasons and reviewed manually
- Determinism validation:
  - regenerate the artifact from the same ordered input and compare stable sections, or at minimum compare source index and principle IDs for unchanged ordering

## Planning Recommendations
- Plan `01-01` to finalize the Markdown schema, source index format, chapter-mode classification convention, and validation checklist before extraction work begins.
- Plan `01-02` to process transcripts in batches, continuously updating the source index and principle catalog, then run the validation checks above and produce the Phase 2 handoff brief.
- Keep the first implementation small and repo-local, likely a script under `scripts/` plus docs under `.planning/phases/01-build-trend-following-knowledge-base/`, matching the project layout in `.planning/codebase/STRUCTURE.md`.
